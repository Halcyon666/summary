Schema based AOP Configuration

For the relevant code fromOfficial Source, please see Spring-test org. Spring frame.mylearntest package.

Schema based AOP Configuration​

Schema-based AOP is a new AOP usage that was added after Spring2.0.The AOP, based on Schema, can be seen from the following two perspectives.

  • When the configuration changes ： Spring framework has been upgraded from 1.x version to 2.x versions, promoting the container configuration from the Dtd-based xml to the Schema-based xml, further improving the flexibility and scalability of the configuration.At the same time, the new Schema is configured in such a way as to provide a unique namespace for Spring’s AOP functionality.The original DTD-based AOP configuration in 1.x can be transposed to Schema-based AOP with a slight conversion, so for this reason the schema-based AOP is only a change in configuration.

  • In order to use AOP form @AspectJ, the same： is required to use Java 5 or higher version of JDK or JRE because the annotation is a feature that was introduced after Java 5 was published.If we have to use the previous version of Java 5 and want to use POJO-based Aspect Statement, we can use the Schema-based Spring AOP.Using Schema-based AOP, we can still use POJO to declare Aspects and related Advice.However, no annotation is required. Configuration can be made directly through Schema's configuration file. Pointcut expression in @AspectJ form can also be configured into Schema-based configuration.

Schema based AOP Configuration​

The new schema-based AOP configuration provides separate configuration elements for Pointcut, Advisor, and Aspect concepts, all of which are contained in a single config element, i.e. <AOP:config/>It has only one attribute, proxy-target-class, proxyTargetClasses properties in ProxyConfig, where we can control whether we use interfaces or class-based proxyTargetClasses.It can have three children inside of <AOP:pointcut>,<AOP:advisor>and<AOP:aspect>, which must be configured sequentially.

For <AOP:config>, the bottom layer is basically achieved using the automatic proxy mechanism in 1.x.The corresponding automatic agent implementation class obtains the necessary knife information based on the internal points of point, advisor, and the subelements of Aspect, and then automatically proxies the bean registered in the container.So, if you want not to <AOP:config>, you can still use AutoProxyCreator to implement the class.

Schema based AOP Configuration​

  1. Simple Migration

The 1.x version of Spring AOP encapsulates cross-cutting concerns through the concept of Advisor. After registering the corresponding Pointcut definition and Advice definition to the container (usually in an XML configuration file based on DTD), it is implemented by declaring the corresponding Advisor. , these Pointcut And the Advisor definition is assembled together, and finally the final weaving is done through an AutoProxyCreator.

After turning to Schema-based configuration in version 2.x, these concepts are in fact the same and the only thing that needs to be changed is a change in the specific configuration.Now use <AOP:advisor> instead of the bean definition declaration of the specific advisory class, uses <AOP:config>instead of AutoProxyCreator.

Use <AOP:advisor> in <AOP:config>to configure the corresponding Advisor, that is to say, Spring AOP's Aspect.

  • id：specifies the flag id for the current Advisor definition
  • pointcut-ref：specifies by this attribute what the current Advisor has to define what the point of view is. Specify the specific point reference registered in the container
  • advice-ref：specifies the reference to Advice's object of the current Advisor
  • The order：specifies the order number of the current Advisor, since all the advisory implementation implements Ordered interfaces

package org.springframework.mylearntest.aop2.schemaapsect;

import org.apache.commons.lang3.exception.ExceptionUtils;
import org.apache.commons.logging.Log;
import org.apache.commons.logging.LogFactory;
import org.aspectj.lang.JoinPoint;
import org.aspectj.lang.ProceedingJoinPoint;
import org.springframework.util.StopWatch;

/**
 * @Author: WhaleFall541
 * @Date: 2020/8/12 21:33
 */
public class SchemaBasedAspect {
	public void method1() {}
	public void method2() {}

	private  final Log logger = LogFactory.getLog(SchemaBasedAspect.class);

	public void doBefore(JoinPoint jp) {
		if (logger.isInfoEnabled()) {
			logger.info("before method {" + jp.getSignature().getName()+"} execution");
		}
	}

	public void doAfterReturning(JoinPoint jp, Object retValue) {
		if (logger.isInfoEnabled()) {
			logger.info("before method {" + jp.getSignature().getName()+"} execution");
			logger.info("with return value: " + retValue);
		}
	}

	public void doAfterThrowing(RuntimeException e) {
		logger.error(ExceptionUtils.getStackTrace(e));
	}

	public void doAfter() {
		logger.warn("release system resources ,etc.");
	}

	// NOTE：第一个参数必须为ProceedingJoinPoint
	public Object doAround(ProceedingJoinPoint pjp) throws Throwable {
		StopWatch watch = new StopWatch();
		try {
			watch.start();
			return pjp.proceed();
		} finally {
			watch.stop();
			if (logger.isInfoEnabled()) {
				logger.info(watch);
			}
		}
	}
}

AOP Use Scene​

  1. Exceptions processed
  • Unchecked exception: java.lang. Error, java.lang.Runtime Exception and its subcategories.The compiler does not check these types during the compilation period.
  • Checked exception: java.lang.Exception and its subcategories, but remove the Runtime Exception branch.These exceptions must be processed and the compiler checks them during the compilation period

![](https://s2.loli.net/2023/07/15/bXHyAk8 FerdfB.png)

When unchecked exception is likely to be dropped in many parts of the system, we can add an exception logic to handle it separately at the top of the call (log recording, notification of the corresponding person).We can achieve a Fair that responds to Faulthandling, allowing it to provide a unified treatment of all possible Faults in the system.This full-time Aspect, which we call Fault Barrier.

  1. Security checks

Initially developed independently of Spring, the Acegi Framework has now been merged into Spring Portfolio and renamed Spring Security.

  1. Cache

In order to avoid the need for additional caches to achieve logic in business logic, we can allow caching to be achieved independently of the achievement of the business object, encapsulated the need for caching in the system through AOP's Aspect, which is woven into only if there is a certain point in the system that requires caching support.

Available Caching products implement EhCache, JBossCache etc.; the Spring Modules project provides integration of existing Caching products so that Caching support can be added to Joinpoint in the system through external declarations.

Define a AspectJ for performance check

PerformanceTraceAspect1
@Aspect
public class PerformanceTraceAspect1 {
    private final Log logger = LogFactory.getLog(PerformanceTraceAspect1.class);

    @Pointcut("execution(public void *.method1())")
    public void method1() {}

    @Pointcut("execution(public void *.method2())")
    public void method2() {}

    @Pointcut("method1() || method2()")
    public void compositePointcut() {}

    @Around("compositePointcut()")
    public Object performanceTrace(ProceedingJoinPoint pjp) throws Throwable {
        StopWatch watch = new StopWatch();
        try {
            watch.start();
            return pjp.proceed();
        } finally {
            watch.stop();
            System.out.println("PT in method" + pjp.getSignature().getName() + "]>>>>>" + watch.toString());
            if (logger.isInfoEnabled()) {
                logger.info("PT in method" + pjp.getSignature().getName() + "]>>>>>" + watch.toString());
            }
        }
    }
}

  1. Solution

When the target object depends on itself, we can also try to make the target object's proxies publicly available and solve the problem of internal calls that are not blocked, as long as the target object is allowed to use the corresponding method on its own proxy.

Spring AOP provides AOPContext to disclose proxy objects of the current target object, so we can get the proxy objects for the current target object by using AOPContext.currentProxy() for the target.and set weaver.setExposeProxy(true); in the test class.
