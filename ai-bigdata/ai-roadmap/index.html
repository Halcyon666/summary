AI RoadMap

基于目标岗位要求的实战导向学习计划

你的技术优势​

  • ✅ 5年 Java 开发经验 + React/Vue 前端能力
  • ✅ Docker 容器化经验
  • ✅ LangChain + Ollama 本地 LLM 使用经验

这意味着你已经具备了 AI 应用工程化 的基础，现在需要补齐的是：Python AI 生态系统 + 向量数据库与知识图谱 + Agent 架构。

下面是一个 3 个月冲刺计划，按优先级排列。

阶段一：Python AI 工具栈速成（第 1-2 周）​

核心任务：从 Java 转向 Python 数据处理​

你已经有编程基础，重点是 快速掌握 Python AI 生态的惯用法。

必学工具

  • Pandas：数据清洗、转换、分组聚合（对标 Java Stream API）
  • NumPy：向量化计算、数组操作（替代 Java 循环）
  • Jupyter Notebook：交互式开发环境

实战项目

  1. 用 Pandas 处理一个 CSV 数据集（比如用户行为日志）

    • 数据清洗：去重、填充缺失值
    • 特征工程：时间窗口统计、分组聚合
    • 输出结果用于后续 RAG 系统

  2. 用 NumPy 实现一个简单的向量相似度计算（余弦相似度）

    • 这是向量数据库的底层原理

学习资源

  • Pandas 官方 10 分钟教程
  • NumPy quickstart
  • 不需要看完整课程，边用边查文档即可

阶段二：向量数据库 + RAG 系统（第 3-4 周）​

这是岗位的核心要求，必须有实战经验​

向量数据库选型 优先级从高到低：

  1. Milvus（岗位明确要求，开源、生产级、支持混合检索）
  2. Supabase（PostgreSQL + pgvector，便于快速原型）
  3. Pinecone（云服务，备选）

关键技能

  • Embedding 生成（使用 sentence-transformers 或 Ollama 本地模型）
  • 向量存储与检索（HNSW 索引原理）
  • 混合检索（向量 + 关键词，提升召回率）
  • 分块策略（chunk size、overlap 对检索质量的影响）

实战项目：构建一个文档问答系统

  1. 数据准备：收集 50-100 篇技术文档（Markdown/PDF）
  2. 文本分块：用 LangChain 的 RecursiveCharacterTextSplitter
  3. 向量化：用 Ollama 的 nomic-embed-text 生成 Embedding
  4. 存入 Milvus：创建 collection，插入向量
  5. 检索测试：实现 Top-K 相似度搜索
  6. RAG 集成：检索结果 + 提示词模板 → LLM 生成答案

评估指标

  • 检索准确率（人工标注 20 个问题，计算 precision@5）
  • 响应延迟（目标 <2 秒）

Docker 部署 用 Docker Compose 编排：

services:
  milvus-standalone:
    image: milvusdb/milvus:latest
  ollama:
    image: ollama/ollama
  your-rag-api:
    build: .

阶段三：Agent 架构深度实践（第 5-7 周）​

岗位要求的核心框架全覆盖​

学习路径（按复杂度递增）

1. ReAct 模式（基础）​

  • 原理：Reasoning（推理） + Acting（执行）循环
  • 实现一个简单的工具调用 Agent
    • 工具：搜索引擎、计算器、数据库查询
    • 让 LLM 自己决定何时调用哪个工具

2. LangGraph（重点掌握）​

  • 为什么重要：状态管理 + 复杂工作流
  • 核心概念：
    • StateGraph：节点 + 边的状态机
    • Checkpointer：保存中间状态（支持暂停/恢复）
    • Human-in-the-loop：关键决策点等待人工确认
  • 实战项目：构建一个多步骤数据分析 Agent
    1. 理解用户查询意图
    2. 决定是查询数据库还是搜索文档
    3. 执行查询
    4. 生成可视化代码（返回 Python matplotlib 代码）
    5. 用户确认后执行

3. AutoGen / CrewAI（多 Agent 协作）​

  • AutoGen：微软出品，适合对话式多轮交互
  • CrewAI：角色分工明确的团队协作
  • 选其一深入：
    • 实现一个 "研究员 + 写作者 + 审核者" 三角色系统
    • 研究员搜集资料 → 写作者生成报告 → 审核者检查质量

4. Dify / LangFlow（低代码平台）​

  • Dify：开源的 LLM 应用开发平台
  • LangFlow：可视化 LangChain 工作流
  • 目标：了解企业级 Agent 平台的架构
    • Prompt 版本管理
    • API 接口自动生成
    • 多模型切换
  • 用 Docker 本地部署 Dify，体验完整流程

5. MCP (Model Context Protocol)​

  • 最新标准：让 LLM 连接外部数据源
  • 实现一个 MCP server：
    • 暴露你的数据库或 API 给 LLM
    • 支持工具发现（tool discovery）
    • 安全认证与权限控制

阶段四：知识图谱入门（第 8-9 周）​

岗位要求的差异化技能​

核心概念

  • 实体（Entity）、关系（Relationship）、属性（Property）
  • 三元组：（主体，谓语，客体）
  • 本体建模（Ontology）：定义领域的概念体系

图数据库：Neo4j

为什么选 Neo4j：

  • 原生图存储（不是关系数据库改的）
  • Cypher 查询语言（类似 SQL，但更适合图）
  • 可视化界面友好

实战项目：构建一个公司组织架构图谱

  1. 实体设计

    • 人员（Person）：姓名、职位、入职时间
    • 部门（Department）：名称、预算
    • 项目（Project）：名称、开始时间

  2. 关系设计

    • WORKS_IN（人员 → 部门）
    • MANAGES（人员 → 部门）
    • PARTICIPATES_IN（人员 → 项目）

  3. 数据导入

    • 从 CSV 或数据库导入到 Neo4j
    • 用 Python neo4j 驱动编写脚本

  4. 复杂查询

    • 找出管理超过 10 人的经理
    • 找出参与项目最多的员工
    • 推荐可能认识的同事（二度连接）

  5. RAG + 知识图谱融合

    • 用户提问："张三在哪个项目组?"
    • Agent 决定是查询图数据库还是向量数据库
    • 结合结构化数据（图谱）+ 非结构化数据（文档）

阶段五：前端 AI 应用开发（第 10-11 周）​

发挥你的 React 优势​

WebSocket 实时通信

  • 实现流式响应（Token-by-Token 输出）
  • 后端：FastAPI + WebSocket
  • 前端：React + WebSocket API 或 Socket.io

核心功能实现

  1. 对话界面

    • 消息列表（用户/AI 交替显示）
    • Markdown 渲染（代码高亮、表格）
    • 打字机效果（逐字显示）

  2. 上下文管理

    • 对话历史存储（LocalStorage 或后端）
    • 多会话切换
    • 上下文长度限制提醒

  3. 高级交互

    • 重新生成回答
    • 编辑已发送的消息
    • 代码块一键复制
    • 思维链可视化（展开 Agent 的推理过程）

  4. 文件上传

    • 拖拽上传文档（PDF、Word、TXT）
    • 前端预览 + 后端处理（文本提取）
    • 进度条显示

技术栈

前端：React + TypeScript + TailwindCSS
状态管理：Zustand 或 React Context
WebSocket：原生 API 或 Socket.io-client
后端：FastAPI + LangChain

部署

  • 前端打包 → Nginx 静态托管
  • 后端 API → Docker 容器
  • 反向代理配置（处理 CORS）

阶段六：模型微调与分布式训练（第 12 周+）​

进阶技能，按需深入​

LoRA / QLoRA 微调

  • 原理：冻结基础模型，只训练小部分参数
  • QLoRA：4-bit 量化 + LoRA（显存需求更低）
  • 使用 Hugging Face peft 库
  • 实战：微调 Llama 3 8B 用于特定领域问答
    • 准备数据集（至少 500 条高质量 QA 对）
    • 在单卡 GPU（如 RTX 4090）上训练
    • 评估困惑度（Perplexity）和任务准确率

DeepSpeed 分布式训练

  • 适用场景：多卡并行训练大模型
  • ZeRO 优化器：显存优化技术
  • 配置文件编写（JSON 格式）
  • 这部分可以先了解概念，除非岗位明确要求实操

关键数据源设计能力​

面试必问的架构设计题​

场景：设计一个 AI 客服系统的数据架构

需要整合：

  1. 关系数据库（PostgreSQL）

    • 用户信息、订单记录、产品目录

  2. 向量数据库（Milvus）

    • 产品说明文档、FAQ、历史客服对话

  3. 图数据库（Neo4j）

    • 产品关联关系、用户购买路径、推荐网络

  4. 数据流动

    • 新增产品 → 自动生成 Embedding → 存入 Milvus
    • 用户咨询 → 先查图谱找相关产品 → 再查向量库找文档 → RAG 生成答案

设计要点

  • 数据同步策略（CDC、消息队列）
  • 冷热数据分离
  • 缓存层设计（Redis）
  • 权限与安全（行级权限控制）

面试准备清单​

必须能回答的问题​

  • Embedding 模型如何选择？（维度、性能、多语言支持）
  • RAG 系统的召回率如何优化？
  • LangGraph 和 LangChain Chains 的区别？
  • 向量数据库的索引类型（HNSW、IVF）
  • 知识图谱和向量数据库的使用场景区别
  • Agent 的幻觉问题如何缓解？
  • LoRA 和全量微调的区别
  • WebSocket 和 SSE 的选择依据

必须有的项目经验​

  1. 一个完整的 RAG 系统（带评估指标）
  2. 一个 Agent 应用（最好用 LangGraph）
  3. 一个知识图谱应用（Neo4j）
  4. 一个前后端完整的 AI 对话应用

最后的建议​

不要陷入的坑​

  • ❌ 追求理论完美（论文、数学推导）
  • ❌ 工具收集癖（每个框架都浅尝辄止）
  • ❌ 纯看视频不动手

正确的学习方式​

  • ✅ 以项目为驱动（倒逼学习）
  • ✅ 快速原型 → 迭代优化
  • ✅ 记录踩坑过程（写技术博客）
  • ✅ 参与开源项目（提 PR 到 LangChain/LangGraph）

时间分配建议​

  • 70% 写代码实战
  • 20% 看文档和源码
  • 10% 看教程视频

❌ 传统学习方式的问题：

看完整个课程(10小时) → 开始写项目 → 发现忘了 → 重新看

结果：浪费大量时间，遗忘率高

✅ 正确的方式：

快速浏览(30分钟) → 立即实战 → 遇到问题 → 针对性学习(10分钟) → 继续实战

这个路线图覆盖了你提到的所有岗位要求，按这个节奏推进，3 个月后你将具备完整的 AI 应用开发能力。祝学习顺利！
